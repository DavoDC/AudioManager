Meet AudioManager

A personal music library is not just files in a folder. This is the C# app that keeps it organized.

The Problem It Solves

You download a song. The filename is "track01.mp3". The artist tag says "various". The genre is blank. And you already have 3,000 songs that need to be in exactly the right folder.

Doing this manually every time would take hours. AudioManager does it in seconds.

🎵

You download it. AudioManager handles the rest.

Fix the tags. Rename the file. Route it to the right folder. Validate everything is still clean. Done.

Two Modes, Two Jobs

AudioManager has two completely separate jobs. You pick one from an arrow-key menu when you launch it.

Analysis

Scans your entire music library, checks every tag, and produces a report. Catches problems before they become permanent. Like a health checkup for your music collection.

Integrate

Takes songs from your Downloads/NewMusic folder, fixes their tags, and moves them into the right place in your library. Like a new-employee onboarding process.

💡
Why two modes?

Analysis is safe to run any time - it never moves files. Integrate is the action - it does. Separating them means you can always look before you leap by running Analysis first.

The Cast of Characters

AudioManager is made of specialist components. Each one does one job and hands off to the next - like a well-run assembly line.

🔄
Reflector

Creates an XML snapshot of every song in your library - the mirror that everything else reads from.

📖
Parser

Reads those XML files and loads the metadata into memory as a list of track objects.

✅
LibChecker

Validates every song against library rules. Reports everything that is not right.

🏷️

Cleans messy tags on new songs - genres, featured artists, TCMP compilation flags, and file naming.

📦
MusicIntegrator

Orchestrates the integration pipeline: fix tags, decide routing, move files.

02

The Mirror

AudioManager never reads your MP3 files directly. It reads a lightweight XML copy of them. Here is why.

Your Library's Passport System

Imagine every song in your library had to carry a passport - a small document listing their name, artist, album, and genre. Instead of flipping through thousands of MP3 files every time (slow, risky), AudioManager just reads the passports.

Those passports live in a separate git repo called AudioMirror. One XML file per song.

AudioMirror/ - separate git repo, lives next to AudioManager
AUDIO_MIRROR/ - one XML per song
Artists/Flume/Skin/01 - Never Be Like You.xml - metadata for one track
Artists/Flume/Skin/02 - Say It.xml
Compilations/Now 90/05 - Mr Jones.xml
LastRunInfo.txt - timestamp of last mirror update

The Analysis Pipeline in Code

Here is what actually runs when you choose Analysis. Six lines of C#. Each one is a specialist that takes the previous one's output.

CODE (Program.cs) 
AgeChecker ac = new AgeChecker(forceMirrorRegen);

Reflector r = new Reflector(mirrorPath);

Parser p = new Parser(mirrorPath);

Analyser a = new Analyser(p.audioTags);

LibChecker lc = new LibChecker(p.audioTags);

ReportWriter.Save(captureWriter.ToString(), libCheckerClean);
PLAIN ENGLISH

Check if the mirror XML files are recent (under 7 days) - or force a rebuild if requested

Walk the music library and write one XML file per song (the mirror)

Read all those XML files into memory as a list of track objects

Calculate stats: artist counts, album counts, genre breakdown, etc.

Apply library rules to the track list - flag anything wrong

Write all the output to a timestamped report file

Why Is the Mirror a Git Repo?

Because git makes every change visible. When you integrate new songs, those XMLs get committed. You can see exactly what changed, when, and why.

📋

Audit Trail

Every song addition creates a commit. The git log is a record of exactly when each song entered your library.

🔙

Rollback

If a bad integration corrupts your library state, you can revert the mirror to see what things looked like before.

🔎

Change Detection

The pre-integration gate uses git status to check if the mirror is clean - no uncommitted changes - before allowing new songs in.

🧠
The Age Check

AudioManager only regenerates the mirror if it is older than 7 days. This is set in Constants.cs: TimeSpan.FromDays(7). For day-to-day use, it reuses the existing mirror - saving seconds on startup.

03

Fix First, Then File

The integration pipeline: how a messy downloaded MP3 becomes a properly organized library entry.

Songs in the Wild

Songs you download arrive with whatever metadata the uploader chose to add. That is often nothing useful - or actively wrong.

Typical download

Title: "Mr. Jones (feat. Adam Duritz) [Official Audio]"
Artist: "Counting Crows ft. Adam Duritz"
Genre: "Rock/Pop"
TCMP: not set

After TagFixer

Title: "Mr. Jones"
Artist: "Counting Crows"
Genre: "Rock"
TCMP: set correctly
File: "Counting Crows - Mr. Jones.mp3"

What TagFixer Does

TagFixer runs before a song is filed. It is like a customs agent who checks every item and brings it up to standard before it enters the country.

Parenthetical strip Removes "(feat. X)", "(Official Audio)", "(Remix)" from titles - junk that does not belong in a clean library
Featured artist Extracts "feat. X" from titles into a separate artist tag - so multiple artists are searchable correctly
Genre cleanup Normalizes messy genre strings like "Rock/Pop" or "rock" into clean canonical values like "Rock"
TCMP flag Sets the Compilation ID3 tag - tells media players whether a song is part of a multi-artist compilation
File rename Renames the file to "Artist - Title.mp3" after tags are cleaned - filename matches metadata

The Safety Net: Dry Run

Before any file moves, AudioManager runs a dry run first. You see exactly what it intends to do, then confirm before anything is actually moved.

CODE (Program.cs) 
new MusicIntegrator(dryRun: true, ...);
Console.WriteLine("--- DRY RUN COMPLETE ---");
Console.Write("Proceed with real integration? [y/N]: ");
string answer = Console.ReadLine();
if (answer != "Y")
{
    Console.WriteLine("Cancelled.");
    return;
}
PLAIN ENGLISH

Run the entire integration pipeline, but with pretend mode on - print what would happen but do not move anything

Print a clear divider so you know the simulation is done

Ask: are you happy with this plan?

Read what you type

If you did not type exactly "Y"...

...cancel everything. Nothing moves unless you explicitly confirm.

04

The Librarian

LibChecker enforces the rules. If anything is wrong with your library, it will find it - and the integration gate will not let you proceed until it is fixed.

Quality Control for a Music Library

Think of LibChecker as a building inspector. Every new development needs inspection before people can move in. Every library analysis runs LibChecker before any integration is allowed.

LibChecker does not fix problems - it only reports them. You fix them manually and run Analysis again until it is clean. Only a clean library can receive new songs.

Duplicate detectionFinds songs that appear twice in the library - same title, same artist
Missing tagsFlags songs with blank artist, title, or album fields - cannot be organized without them
Wrong folderCatches songs that ended up in a folder that does not match their tags - like a compilation in the Artists folder
Invalid filenamesFinds filenames that do not follow the "Artist - Title.mp3" convention or contain illegal characters
Orphan XMLsDetects XML mirror files with no matching MP3 - a sign that a song was deleted but the mirror was not updated

The Gate Before the Door

You cannot integrate new songs until the library is clean. This is enforced in code. The pre-integration gate runs a full Analysis before allowing a single file to move.

1
Regenerate the mirror

Forces a fresh Reflector run - ensures the gate sees the real current state of your library

2
Check if the mirror is clean

Runs "git status" on AudioMirror - if there are uncommitted changes, integration is blocked

3
Run LibChecker

If LibChecker finds any issues, integration is blocked. "Fix library issues before adding new songs."

4
Gate passed - proceed

Only if all three checks pass does MusicIntegrator get to run the dry run and real integration

💡
Why enforce this?

Adding songs to a library with existing problems compounds the mess. Fix the foundation first, then build on top of it.

When Things Break

Everything you need to debug a problem - and the architectural tricks that make it possible.

The Dual Output: TeeWriter

Every run of AudioManager creates a log file automatically. You see output in the terminal AND it gets saved to logs/ simultaneously. This is the TeeWriter.

🔀
What is a TeeWriter?

Named after the Unix "tee" command: like a T-junction in a pipe - water flows through both branches at once. TeeWriter intercepts every line of output and writes it to both the screen and a log file simultaneously. You never have to choose between watching live output and having a record.

CODE (Program.cs) 
string logFilePath = Path.Combine(
  Constants.LogsPath,
  $"{logPrefix}-{timestamp}.log"
);
TeeWriter teeWriter = new TeeWriter(
  Console.Out,
  logFilePath,
  captureWriter,
  addFileTimestamps: isIntegrate
);
Console.SetOut(teeWriter);
PLAIN ENGLISH

Build the log file path: "logs/integrate-2026-06-30_143022.log"

(timestamp makes every log file unique - no overwriting)

Create a TeeWriter with three destinations: screen, log file, and an in-memory buffer

(the in-memory buffer captures Analysis output for the report file)

For integration runs, add timestamps to each log line

Replace the console output with TeeWriter - every Console.WriteLine now hits all three destinations

Reading the Logs

When something goes wrong, the log file is your first stop. Log files live in AudioManager/logs/ and are gitignored (they are ephemeral, not history).

🔴

LibChecker errors

Look for lines with "ISSUE:" or "ERROR:". Each one names the problem and the file. Fix the file, run Analysis again, repeat until clean.

WARN

Integration WARN vs INFO

WARN means something unexpected. INFO is normal progress. If you see "WARN: missing from library", it might be a file deleted in the same run - check routing decisions above it.

⏱️

Timing info

Each component prints how long it took. If an Analysis run is slow, the timing tells you exactly which step to investigate - Reflector is usually the culprit on large libraries.
